Description: Investigating Potential WordPress Plug-in Issue

Tool(s) used: WordPress, Browser DevTools, Server Logs

The 5 W's:

  • Who: Myself (Security Analyst)
  • What: Investigating potential issue with WordPress site due to plug-in conflict
  • Where: Company Website / WordPress Environment
  • When: April 26, 2025, 2:00 PM
  • Why: To resolve performance issues and prevent possible security vulnerabilities related to a WordPress plug-in conflict

Additional notes:

  1. Diagnostics Performed: - Ping Test: I used the ping command to check for network connectivity to the server, which confirmed that the server was reachable but the website was still down. This suggested that the issue wasn't network-related. - top Command: Running the top command on the server revealed that MySQL and Redis were consuming significant resources. This indicated that there might be a resource bottleneck caused by either the database or a plug-in. - df -h Command: I used the df -h command to check disk space usage and found that the root directory had 66% usage, which seemed acceptable, but it helped rule out disk space issues.
  2. Initial research indicated that certain WordPress plug-ins can cause site performance issues and errors.
  3. Notified my superior about the potential plug-in conflict affecting the WordPress site’s performance.
  4. Suggested further investigation into deactivating specific plug-ins to isolate the issue.
  5. Plan to verify logs for errors related to WordPress plug-ins and analyze resource consumption to pinpoint problematic plug-ins.

Date: April 23, 2025

Entry: #1

Description: Investigating and responding to suspicious activity potentially linked to website performance issues.

Tool(s) used: AWS, Nginx logs, IP analysis tools, Linux CLI, manual log inspection.

The 5 W's:

  • Who: Multiple unidentified IP addresses, primarily associated with international hosting services—likely automated bots or scanners.
  • What: Suspicious and repeated web traffic from global IPs, leading to site instability and unexplained server behavior.
  • Where: On the production server hosting the website for the company I work for.
  • When: The issue was detected early morning around 5:20 AM on April 23, 2025.
  • Why: The event was triggered by a combination of frequent HTTP requests (some targeting suspicious endpoints) and a series of critical Nginx errors involving cache file handling, coinciding with the website’s downtime. These requests were likely part of a widespread automated scan targeting vulnerable web applications.

Additional notes:

  1. Root Trigger: Sales and marketing lead noticed the website was down and flagged the issue for investigation. I began examining web server logs around the time the site went down. The timeline aligned with a sharp increase in traffic from persistent, globally distributed IPs.
  2. Response: Identified top offending IP addresses (e.g., 78.153.140.156, 1.14.7.100) for potential blocking. Confirmed requests targeting old ASP.NET and WordPress-specific endpoints—neither of which the site supports—indicating generic bot behavior. Recommended blocking repeated offenders via the firewall and updating server rules to ignore or rate-limit suspicious traffic.
  3. Next steps: Possibly implement reverse proxy filtering for legacy probing (ASP.NET, WordPress, etc.). Begin documenting IP threat intelligence and automate blacklist updates.
